# Выбор архитектуры: один запрос vs связи vs объединение

Источник: стр. 13–14, 64–68, 72–77, 325.

## Матрица решений

| Требование | Решение |
|------------|---------|
| Справочник + остатки, простая форма | **Один запрос**, LEFT JOIN |
| Регистр + документ-детализация, разная гранулярность | **Два набора + связь** + `В (&ПараметрСвязи)` в приёмнике |
| Поступление **и** расход — все строки обоих, без сопоставления | **Набор «Объединение»** + два запроса с выровненными полями |
| Поступление **с** расходом по номенклатуре (сопоставление) | **Два набора + связь** (LEFT) или один запрос с JOIN |
| Запрос слишком сложный, итоги по остаткам, смешение запрос+ТЗ | **Несколько наборов**, не один тяжёлый JOIN |

## Объединение (пример 5, стр. 72)

Структура:

```
НаборДанных1 (Объединение)
├── Поступление (запрос к ТЧ ПриходнаяНакладная)
└── Расход (запрос к ТЧ РасходнаяНакладная)
```

Поля выравниваются псевдонимами:

- `Количество` → `Поступило` / `Продано`
- `Сумма` → `СуммаПоступления` / `СуммаПродажи`
- общее: `Номенклатура`

Аналог языка запросов: `ОБЪЕДИНИТЬ ВСЕ` между двумя `ВЫБРАТЬ`.

## Связь (пример 3 vs 5)

- **Связь:** показать цены **и** документы, где документ = регистратор цены (не все документы).
- **Объединение:** анализ поступления **и** продажи как единый список движений (все строки обоих).

## Один запрос vs два набора (совет из прим. 4, стр. 148–153)

Один запрос (JOIN) — когда задача узкая и запрос остаётся читаемым.

Два связанных набора — когда:

- один запрос раздувается;
- нужны итоги без дублирования;
- источники разного типа (запрос + объект).

## Связь наборов = LEFT JOIN между наборами

Не путать с **Объединением** наборов и с `ОБЪЕДИНИТЬ` в языке запросов внутри одного набора.
