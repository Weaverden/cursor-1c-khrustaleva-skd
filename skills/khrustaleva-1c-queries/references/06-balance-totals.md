# Итоги по полям остатка в СКД

Источник: гл. 3, стр. 320–324; пример 14, стр. 167+.

## Роль поля «Остатки»

В схеме СКД полю назначается роль **Остатки** (и связанные роли периодов/измерений). Итоги по таким полям считаются иначе, чем простая `СУММА`.

## Правило уникальности

Для корректных итогов в данных должна соблюдаться **уникальность комбинации полей-периодов и полей-измерений**.

## Ошибка: детализация без родительского периода

**Неправильно** — `Регистратор` без родительского поля-периода:

```bsl
ВЫБРАТЬ
    ...
    ОстаткиНоменклатурыОстаткиИОбороты.Регистратор
ИЗ
    РегистрНакопления.ОстаткиНоменклатуры.ОстаткиИОбороты(, , Авто, , ) КАК ...
```

**Правильно** — добавить родительский период, например `ПериодСекунда`:

```bsl
ВЫБРАТЬ
    ...
    ОстаткиНоменклатурыОстаткиИОбороты.Регистратор,
    ОстаткиНоменклатурыОстаткиИОбороты.ПериодСекунда
ИЗ
    РегистрНакопления.ОстаткиНоменклатуры.ОстаткиИОбороты(, , Авто, , ) КАК ...
```

**Правило:** при выборе поля-периода в запросе должно присутствовать его **родительское** поле-период (для всех уровней иерархии периодов).

## Нумерация периодов

Поля-периоды должны иметь **непрерывную нумерацию с 1**; чем меньше номер — тем **точнее** период.

Пример иерархии:

- `ПериодСекунда` — период 2
- `Регистратор` — период 1 (уточнение секунды)

`НомерСтроки` уточняет `Регистратор` → номер периода `НомерСтроки` **меньше**, чем у `Регистратор`.

При **Автозаполнение** ролей в схеме платформа может проставить роли сама; иначе — вручную.

## Бухгалтерские остатки

Расчёт итогов по полям бух. остатков аналогичен обычным остаткам + учитывается поле-счёт (роль «бухгалтерский остаток»).

## Практика для запросов

1. Для отчётов «движение + остатки с регистратором» всегда включать цепочку периодов из виртуальной таблицы.
2. Не смешивать в одной группировке СКД поля несвязанных наборов при расчёте остатков.
3. Проверять роли полей в схеме после изменения текста запроса.
